# BalloonShoot Project Memo

作成日: 2026-04-08

## 位置づけ

この文書は、要件確定前のプロジェクトメモである。
最終仕様ではなく、今後の要件整理と設計の起点として扱う。

## プロジェクトの目的

放課後等デイサービス向けに、短時間で遊べるシンプルな手遊びゲームを作る。

プレイヤーはノートPCのWebカメラの前で手を動かし、

- 人差し指で照準を動かす
- 親指を使って引き金を引くように発射する
- 画面内に浮かんでくる風船を撃つ

という流れで遊ぶ。

## ゲーム内容の現時点の想定

### コア体験

- カメラに映った片手を使って遊ぶ
- 人差し指先端の位置に対応して照準が動く
- 親指と人差し指の接近を発射入力として扱う
- 風船が下から上へゆっくり浮かぶ
- 風船に照準を合わせて発射すると割れる
- 割れたときに分かりやすい視覚効果と音を返す

### 1プレイの想定

- 1プレイは 30 秒から 90 秒程度
- スコア制にして、成功体験が分かりやすい構成にする
- 難しすぎる失敗ペナルティは入れない
- すぐ再開できる

### 放デイ向けに重視すること

- ルールが一目で分かる
- 誤発射してもストレスになりにくい
- 追従が安定していて、指が少し揺れても遊べる
- 派手すぎず、しかし反応は分かりやすい
- セットアップが簡単で、ローカル環境だけで動く

## このプロジェクトで達成したい目標

### 必須目標

- Webブラウザ上で動作する
- ノートPC内でローカル完結する
- 外部サーバーを使わない
- 一般的なノートPCの内蔵Webカメラで遊べる
- 片手だけで照準移動と発射ができる
- 風船を撃つゲームとして成立する最小限の演出がある

### できれば達成したい目標

- 低スペックでも大きく処理落ちしない
- 子どもが見てすぐ操作を理解できる
- 療育現場で短時間に繰り返し遊べる
- 将来的に難易度調整やモード追加がしやすい構成にする

### 今回はスコープ外とするもの

- オンライン対戦
- ユーザー登録や保存機能
- 高度な3D表現
- 複数人同時プレイ
- モバイル最適化
- 学習記録や分析ダッシュボード

## 技術スタックの方針

### 推奨方針

MVP は、Web + MediaPipe + Canvas 2D で構成する。

理由:

- ローカル完結しやすい
- ノートPCで動かしやすい
- 実装規模を抑えられる
- まずゲーム性検証に集中できる

### 推奨スタック

- 実行環境: モダンブラウザ
- 入力: Webカメラ
- 手の追跡: MediaPipe Hand Landmarker
- 描画: HTML Canvas 2D
- 実装言語: TypeScript
- 開発基盤: Vite
- 音: ブラウザ標準の audio 再生、または軽量な効果音ファイル

### 役割分担

- MediaPipe: 手のランドマーク取得
- ゲームロジック: 照準、発射、風船生成、当たり判定、スコア
- 描画層: 背景、照準、風船、UI、エフェクト

## 入力仕様の現時点の想定

### 照準

- 人差し指先端の座標を照準位置として使う
- そのままだと揺れやすいため、平滑化をかける
- カメラ映像と画面座標の対応は直感的になるよう調整する

### 発射

- 親指先端と人差し指先端の距離で発射判定を行う
- いわゆる pinch に近い判定を使う
- 誤発射防止のため、押下と解放で別の閾値を持たせる
- 手の大きさに依存しすぎないよう、距離は正規化して扱う

### 安定化のために必要な処理

- 座標の平滑化
- 発射判定のデバウンス
- 手が一瞬見えなくなったときの復帰処理
- 片手のみを対象にする制御

## 想定アーキテクチャ

大きく以下の単位に分ける。

- Hand Tracking Layer
  - カメラ入力の取得
  - MediaPipe 実行
  - 指先座標と発射状態の算出

- Input Mapping Layer
  - ランドマークからゲーム入力への変換
  - 照準位置の補正
  - 発射イベントの生成

- Game Layer
  - 風船の生成と移動
  - 当たり判定
  - スコア
  - 制限時間やゲーム状態

- Presentation Layer
  - Canvas 描画
  - UI 表示
  - エフェクト
  - 音

この分離により、将来的に入力方式を差し替えやすくする。

## 想定するMVP

最初に目指すMVPは次の内容とする。

- カメラ起動
- 片手検出
- 照準表示
- pinch による発射
- 風船が一定間隔で出現
- 命中で風船が割れる
- スコアが増える
- 制限時間終了で結果表示
- リトライできる

## 性能と運用の前提

- 一般的なノートPCで動作することを優先する
- 初期実装では 640x480 程度のカメラ入力を想定する
- まずは 1 人、1 手、1 画面で成立させる
- すべてローカル実行とし、映像やデータを外部送信しない

## 現時点の主なリスク

- 内蔵カメラや照明条件によって追跡精度がぶれる
- 指先座標をそのまま使うと照準が揺れやすい
- 発射判定が敏感すぎると誤発射が増える
- CPU負荷が高いとゲーム描画と推論が競合する
- 利用者ごとの手の大きさや姿勢の差で操作感が変わる

## 次に決めるべき要件

次の論点を順に確定する。

1. 1プレイの長さとゲームオーバー条件
2. 難易度の基本方針
3. 風船の見た目と演出の方向性
4. 音の有無と強さ
5. UI 表示の最小構成
6. 発射判定の体験を pinch に寄せるか、別ジェスチャーにするか
7. 対象ブラウザの範囲

## 現時点の推奨判断

現段階では、以下の方針が最も現実的と考える。

- Webアプリとして作る
- Canvas 2D で軽量に描画する
- MediaPipe Hand Landmarker を片手追跡で使う
- まずは最小限の2Dゲームとして成立させる
- その後、必要なら演出や難易度を追加する
